## 7) Potential threading issues in Avalonia projects

These are **potential risks** identified by code inspection (not fixed in this pass):

1. **UI-bound properties potentially updated off UI thread**
   - `src/Arbor.HttpClient.Desktop/Features/Sse/SseViewModel.cs`
   - `src/Arbor.HttpClient.Desktop/Features/WebSocket/WebSocketViewModel.cs`
   - Both use `ConfigureAwait(false)` in connection flows and then assign observable/UI-bound properties outside explicit UI-thread marshaling in some paths (`catch`/`finally`).

2. **Fire-and-forget background task touching UI state**
   - `src/Arbor.HttpClient.Desktop/Features/Main/MainWindowViewModel.cs` (`OnRequestBodyFileChanged`)
   - Uses `Task.Run` + delayed read + dispatcher update with broad exception swallow; possible lifecycle race during shutdown/dispose.

3. **Scheduled job start race window**
   - `src/Arbor.HttpClient.Desktop/Features/ScheduledJobs/ScheduledJobService.cs`
   - `ContainsKey` + assignment on `ConcurrentDictionary` is not atomic as a combined “start-if-not-running” operation; concurrent starts may double-schedule.

4. **Known deadlock hazard documented and mitigated**
   - `src/Arbor.HttpClient.Desktop/Features/HttpRequest/RequestView.axaml.cs` includes explicit deferred-dispatch mitigation comments around TextMate grammar application.
   - This is a positive sign of active threading-risk handling.

### 1. Integration tests are not run in CI
**File:** `.github/workflows/ci.yml`

`ci.yml` runs `Arbor.HttpClient.Core.Tests` and the E2E tests but never executes `Arbor.HttpClient.Core.Integration.Tests`.  
`docs/coverage.md` documents coverage numbers from those Kestrel-based WebSocket/SSE tests, but a broken integration test would not block a PR.

**Fix:**  
Add an `integration-tests` job (after `build`) to `ci.yml`:
```yaml
- name: Run integration tests
  run: >
    dotnet test src/Arbor.HttpClient.Core.Integration.Tests/Arbor.HttpClient.Core.Integration.Tests.csproj
    --no-restore
    --configuration Release
    --logger "console;verbosity=normal"
```

### 2. SQLite repository tests are in the wrong test project
**File:** `src/Arbor.HttpClient.Core.Tests/SqliteRepositoriesTests.cs`

`SqliteRepositoriesTests` tests types from `Arbor.HttpClient.Storage.Sqlite`, but it lives in `Arbor.HttpClient.Core.Tests`.  
`docs/architecture/clean-feature-separation.md` explicitly requires: *"Tests for a library project must live in a test project that references only that library plus `Arbor.HttpClient.Testing`."*  
Currently `Core.Tests` carries an indirect reference to the Storage project, violating layer boundaries.

**Fix:**  
1. Create `src/Arbor.HttpClient.Storage.Sqlite.Tests/` with a new `.csproj` referencing `Arbor.HttpClient.Storage.Sqlite` and `Arbor.HttpClient.Testing`.  
2. Move `SqliteRepositoriesTests.cs` (and any other storage-layer tests) to the new project.  
3. Remove the `<ProjectReference>` to `Arbor.HttpClient.Storage.Sqlite` from `Arbor.HttpClient.Core.Tests.csproj`.  
4. Add the new test project to `Arbor.HttpClient.slnx`.

### 3. README documents the wrong behavior for unresolved variables
**File:** `README.md`

The README states:
> *"If a variable name has no match in the active environment it is left unchanged in the outgoing request."*

`VariableResolver.Resolve` actually replaces unmatched `{{token}}` with an **empty string**, and the test suite explicitly asserts this (`Resolve_ShouldReplaceUnknownTokenWithEmptyString`, `Resolve_UndefinedVariableInUrl_ShouldEvaluateToEmptyString`).

**Fix:**  
Update the README sentence to:
> *"If a variable name has no match in the active environment, the `{{token}}` placeholder is replaced with an empty string."*

### 4. Silent exception swallowing in two places
**Files:** `src/Arbor.HttpClient.Desktop/App.axaml.cs`, `src/Arbor.HttpClient.Desktop/ViewModels/MainWindowViewModel.cs`

```csharp
// App.axaml.cs â€” after Log.Logger is fully configured
try { currentOptions = optionsStore.Load(); }
catch { currentOptions = new ApplicationOptions(); }   // swallows everything silently

// MainWindowViewModel.PersistLayoutOptions
catch { /* best-effort persistence for implicit layout save */ }
```

Both patterns use a bare `catch` with no logging. Errors in options loading or layout persistence are invisible in production, making user-reported issues very hard to diagnose.

**Fix:**  
Replace each bare `catch` with `catch (Exception ex)` and add a `Warning`-level log entry:
```csharp
catch (Exception ex)
{
    _logger.Warning(ex, "Failed to load application options; using defaults");
    currentOptions = new ApplicationOptions();
}
```

### 5. TLS 1.0 and 1.1 are selectable without any warning
**Files:** `src/Arbor.HttpClient.Desktop/ViewModels/MainWindowViewModel.cs`, `src/Arbor.HttpClient.Desktop/App.axaml.cs`

`TlsVersionOptions` exposes `"Tls10"` and `"Tls11"`. Both protocols are cryptographically broken and cannot connect to any modern server. Selecting them is silent â€” no UI warning, no log event.

**Fix (choose one):**
- **Preferred:** Remove `"Tls10"` and `"Tls11"` from `TlsVersionOptions`. They serve no practical purpose against modern endpoints.
- **Alternative:** Keep them but display a dismissible `InfoBar`/banner in the Options panel when either is selected, and log a `Warning` message explaining why the selection is unsafe.

### 6. Response-formatting utility methods belong in a dedicated class, not `MainWindowViewModel`
**File:** `src/Arbor.HttpClient.Desktop/ViewModels/MainWindowViewModel.cs`

The view model contains ~150 lines of static pure functions:

**Fix:**

### 7. `OptionsViewModel` holds a reference to `MainWindowViewModel`
**File:** `src/Arbor.HttpClient.Desktop/ViewModels/MainWindowViewModel.cs`, line ~348

```csharp
_optionsViewModel = new OptionsViewModel(this);
```

This creates a tight circular dependency. The architecture doc orders extracting feature VMs with minimal contracts; `OptionsViewModel` should receive only the specific properties and commands it binds to rather than the whole main VM.

### 8. `HttpRequestService` has racy mutable post-construction state
**File:** `src/Arbor.HttpClient.Core/Services/HttpRequestService.cs`

`SetHttpClientFactory`, `SetHttpDiagnosticsEnabled`, `SetHttpDiagnosticsObserver` mutate private fields after construction. If a scheduled job calls `SendAsync` concurrently while the user saves options (which calls `Set*` from the UI thread), there is an unsynchronized data race on `_httpClientFactory`, `_httpDiagnosticsEnabled`, and `_diagnosticsObserver`.

**Fix:**  
Accept these values as constructor parameters, or wrap them in a single `HttpRequestServiceOptions` record that can be replaced atomically (via `Interlocked.Exchange` on a reference, or by recreating the service on options change).

### 9. `VariableResolver.Resolve` is O(n Ă— m)
**File:** `src/Arbor.HttpClient.Core/Services/VariableResolver.cs`

Inside the `Regex.Replace` callback there is a `foreach (var variable in variables)` linear scan. For each `{{token}}` occurrence in the input, all `m` variables are scanned. A request with 10 tokens and 50 variables performs 500 comparisons on every resolve call (which happens on every keystroke in the preview).

**Fix:**  
Build a `Dictionary<string, string>` from `variables` once before calling `Replace` and do O(1) lookups inside the callback:
```csharp
var lookup = variables.ToDictionary(v => v.Name, v => v.Value, StringComparer.OrdinalIgnoreCase);
return TokenPattern.Replace(input, match =>
{
    var key = match.Groups[1].Value.Trim();
    return lookup.TryGetValue(key, out var value) ? value : string.Empty;
});
```

### 10. `SavedRequest` model does not persist headers â€” "Copy as cURL" from history loses them
**Files:** `src/Arbor.HttpClient.Core/Models/SavedRequest.cs`, `src/Arbor.HttpClient.Storage.Sqlite/SqliteRequestHistoryRepository.cs`, `src/Arbor.HttpClient.Core/Services/CurlFormatter.cs`

`CopyHistoryItemAsCurlAsync` calls `CurlFormatter.Format(request)` on a `SavedRequest`. `SavedRequest` has no `Headers` property, so `Authorization` headers and other custom headers set in the original request are silently dropped in the generated cURL command.

**Fix:**  
1. Add `IReadOnlyList<RequestHeader>? Headers` to `SavedRequest`.  
2. Add a `headers` TEXT column (JSON array) to the `request_history` SQLite table and update `SaveAsync` / `GetRecentAsync`.  
3. Populate the field in `HttpRequestService.SendAsync` from the outgoing request's headers.  
4. Update `CurlFormatter` to include the extra headers.

### 17. SQLite request history grows unboundedly
**File:** `src/Arbor.HttpClient.Storage.Sqlite/SqliteRequestHistoryRepository.cs`

Every `SendAsync` call inserts a row. There is no trim/evict query to limit the table size. Over time on a frequently-used machine the database will grow without bound.

**Fix:**  
After each `INSERT`, run a trim query:
```sql
DELETE FROM request_history
WHERE id NOT IN (
    SELECT id FROM request_history
    ORDER BY created_at_utc DESC
    LIMIT 500
);
```
Expose the limit (500) as a configurable setting in `ApplicationOptions`.
